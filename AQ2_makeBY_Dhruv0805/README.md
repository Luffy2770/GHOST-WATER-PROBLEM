# AquaWatch — Ahmedabad NRW Monitor

Real-time Non-Revenue Water detection dashboard for Ahmedabad's water
distribution network. Flask backend + Leaflet map + IoT telemetry simulator.

---

## Folder Structure

```
aquawatch/
├── app.py                  ← Flask server (all API endpoints + auth)
├── generate_network.py     ← Run once to create network.json
├── iot_simulator.py        ← Run alongside app.py (writes live data)
├── whatsapp.py             ← Twilio WhatsApp dispatch helper
├── requirements.txt        ← Python dependencies
├── .env.example            ← Copy to .env and fill in secrets
│
├── templates/
│   ├── index.html          ← Main dashboard (map + alerts + dispatch)
│   ├── login.html          ← Login page
│   └── signup.html         ← Signup page
│
├── network.json            ← Generated by generate_network.py
├── network_live.db         ← Created automatically by iot_simulator.py
├── users.db                ← Created automatically on first run
├── dispatches.json         ← Created automatically on first dispatch
├── model.pkl               ← YOUR ML model (place here)
└── TS-PS10.xlsx            ← YOUR dataset (place here)
```

---

## Setup (First Time)

### 1. Place your files
Copy these into the `aquawatch/` folder:
- `model.pkl`    — your trained anomaly + classifier bundle
- `TS-PS10.xlsx` — your sensor dataset

### 2. Install dependencies
```bash
cd aquawatch
pip install -r requirements.txt
```

### 3. Configure environment
```bash
cp .env.example .env
# Edit .env and set your FLASK_SECRET_KEY at minimum
# Add Twilio credentials only if you want WhatsApp alerts
```

### 4. Generate the network topology (ONCE)
```bash
python generate_network.py
```
This creates `network.json` with 150 segments (50 per zone) that the
map uses to draw real tree-based pipelines.

---

## Running the App

Open **3 terminals** in the `aquawatch/` folder:

**Terminal 1 — IoT Simulator** (writes live sensor data every 10s)
```bash
python iot_simulator.py
```

**Terminal 2 — Flask Server**
```bash
python app.py
```

**Terminal 3 — (optional) watch logs**
```bash
# Nothing needed — just watch the other two terminals
```

Then open: **http://localhost:5000**

## API Endpoints

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/` | Main dashboard (auth required) |
| GET | `/network-topology` | Network JSON for map rendering |
| GET | `/live` | Latest telemetry per segment |
| GET | `/stats` | Summary stats (anomalies, loss, etc.) |
| GET | `/alerts` | All anomaly rows |
| GET | `/zone-summary` | Loss breakdown by zone |
| POST | `/predict` | Run ML prediction on sensor input |
| POST | `/dispatch` | Send WhatsApp work order |
| POST | `/confirm` | Twilio webhook for crew confirmation |
| GET | `/work-orders` | Dispatch log |
| POST | `/false-positive` | Mark segment as false positive |

---

## Data Flow

```
generate_network.py → network.json
        ↓
iot_simulator.py → network_live.db (every 10s)
        ↓
app.py reads network_live.db → /live, /stats, /alerts
        ↓
index.html fetches endpoints → renders map + alerts
```

When `network_live.db` exists, all endpoints use live IoT data.
When it doesn't (e.g. simulator not started), endpoints fall back
to the Excel dataset (TS-PS10.xlsx).

---

## WhatsApp Dispatch

Without Twilio configured, dispatch still works in **stub mode**:
- Work orders are logged to `dispatches.json`
- No actual SMS is sent
- The dashboard shows "Dispatched" status normally

To enable real WhatsApp messages:
1. Get Twilio credentials from https://console.twilio.com
2. Add them to `.env`
3. Set crew phone numbers for each zone in `.env`

---

## Notes

- The map polling interval is **10 seconds** — aligned with the IoT
  simulator's write interval so you see updates in real time.
- The fallback synthetic grid activates if `/network-topology` fails,
  so the map always renders something.
- `users.db`, `network_live.db`, and `dispatches.json` are all
  auto-created on first run — you don't need to create them manually.
